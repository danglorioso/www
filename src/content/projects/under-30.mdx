## The Story

In the Tufts University CS Department, there's a hard rule: **functions must be no longer than 30 lines.** The idea is simple — a function that fits on a single screen is easier to read, easier to test, and easier to debug. Break it into smaller pieces if it grows.

The problem is enforcing it. Every assignment submission, I'd manually count lines to make sure my functions were compliant before submitting. It was tedious and error-prone. Other students would lose points on style because a function crept to 32 or 35 lines without them noticing.

So in May 2024, I built a VS Code extension to automate it. What started as a personal script to stop losing track of line counts turned into a published extension on the VS Code Marketplace — and eventually crossed **800 downloads**.

## What It Does

Under 30 adds two things to your VS Code environment:

1. **A live line counter in the status bar** — select any range of lines and the count shows up immediately, no matter the language or file type.
2. **A function length scanner** — on demand, it scans every function in the file and flags any that exceed the configured limit, reporting the exact start and end line numbers of each violation.

It also checks for unbalanced braces before scanning, since mismatched `{` and `}` would produce garbage results and usually indicate a structural bug anyway.

## How the Detection Works

The function scanner is brace-based rather than AST-based. It tracks the depth of `{` and `}` characters as it walks the file line by line, counting lines between the opening and closing brace of each top-level function.

This approach has a few key advantages:
- **Language-agnostic** — the same logic works for JavaScript, TypeScript, Java, C++, and any other curly-brace language without needing a separate parser per language.
- **Handles nesting correctly** — inner blocks (`if`, `for`, `while`) don't confuse the scanner because it tracks brace depth, not just the first `}` it finds.
- **Validates structure first** — unbalanced braces are caught before counting begins, so warnings are always accurate.

The tradeoff: it won't work for languages that use indentation instead of braces (Python, Ruby). But for the CS department's assignments — all C and C++ — it was exactly the right tool.
